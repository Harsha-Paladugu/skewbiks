Skewbiks.com (like the upstream Pyraminx.net) is a **static, no-build-required website**. Five HTML pages share one engine and a set of UI layers. The only build step compiles the algorithm data and bundles the React trainer; everything else runs as plain `<script>` tags in the browser.

## Commands

```bash
npm install
npm run build         # full pipeline: build:trainer (sheet + trainer bundle), then stamp asset hashes, then npm run check
npm run build:trainer # build:sheet, then esbuild-bundle the trainer (node build.mjs) → js/trainer.js
npm run build:sheet   # compile data/skewb_algs.json → js/sheet.js + data/classmap.json
npm run watch:trainer # esbuild watch on src/trainer (node build.mjs --watch) — does NOT recompile the sheet
npm run stamp         # rewrite each asset's ?v= query to its content hash (tools/stamp-assets.mjs)
npm run check         # verify the shipped js/sheet.js against the engine (alias: npm test)
npm run check:fresh   # re-run the pipeline; fail if js/sheet.js, js/trainer.js or any HTML stamp is stale
npm run test:engine   # engine unit tests (tools/test-engine.mjs)
npm run test:trainer  # trainer substrate tests (tools/test-trainer.mjs) — builds TWO full BFS tables, slow-ish
npm run test:rules    # Firestore rules emulator tests (opt-in: needs the emulator + dev deps)

node tools/compile-sheet.mjs --check   # dry-run compile + report; never writes js/sheet.js
```

The esbuild step lives in `build.mjs` at the repo root (entry `src/trainer/index.jsx` → `js/trainer.js`); there is no separate `build:trainer`-only esbuild script — `node build.mjs` is it.

- There is **no unit-test framework**, but three dependency-light test runners exist. `npm test` is aliased to `tools/check-sheet.mjs` (validates the compiled data sheet only). `npm run test:engine` (`tools/test-engine.mjs`) asserts engine invariants — alg parse/notation, mirror/inverse symmetry, `realCanonKey` stability, the optimal BFS solver. `npm run test:trainer` (`tools/test-trainer.mjs`) asserts the trainer substrate (`src/trainer/skewb-core.mjs`) — case model over the JSON, presentation synthesis, masked scrambles, the first-layer predicate/goal tables. `npm run test:rules` (`test/firestore.rules.test.mjs`) is opt-in and needs the Firebase emulator + dev deps. To exercise the engine ad-hoc in Node, replicate what the tools do: `globalThis.window = {}; require('./js/engine.js'); const E = globalThis.window.OOEngine;` (see the "module strategy" gotcha below).
- **To preview the site locally you must serve over HTTP** (e.g. `npx serve` or `python -m http.server`), not `file://` — `js/account.js` uses dynamic `import()` and the OO/Algorithms pages `fetch` the JSON, both of which need an origin.

## Generated files — do not hand-edit

- **`js/sheet.js`** and **`data/classmap.json`** are generated by `tools/compile-sheet.mjs` from `data/skewb_algs.json`.
- **`js/trainer.js`** is the minified esbuild bundle of `src/trainer/` (the source of truth is `src/trainer/skewb-trainer.jsx` + `skewb-core.mjs`; `js/trainer.js` is build output, even though it's the file an editor may surface).

Both are committed so the site works on the host without a build. Edit the *sources*, then rebuild.

## Cache-busting discipline

Every local `js/`/`css/`/`img` asset is loaded with a `?v=<hash>` query in the HTML, where `<hash>` is the first 8 hex of the asset's content sha1. **`npm run stamp` (`tools/stamp-assets.mjs`) rewrites these from the file bytes and is part of `npm run build`** — editing an asset and rebuilding updates every page that loads it automatically; there is no manual `?v=N` to bump. `npm run check:fresh` (`tools/check-fresh.mjs`) re-runs the whole pipeline and fails if the committed `js/sheet.js`, `js/trainer.js`, or any HTML stamp is stale — wire it into CI / run it before committing. (Note: `?v=` queries embedded in CSS `url()`s, e.g. `img/logo.svg` in `css/site.css`, are not stamped — only HTML refs are.)

## Architecture

### Data flow (single authoring source)

```
data/skewb_algs.json   ← authored algorithm list (the authority; v1 = imported method sheets)
   │ npm run build:sheet                          ↖ fetched at runtime by algs.html AND trainer.html
   ▼
js/sheet.js  (SHEET = { ALG / NAME / CNAME / PRES })  +  data/classmap.json   ← generated data-quality gates; no page consumes them at runtime
```

The **Algorithms page, the trainer AND the solver read `skewb_algs.json` directly** (fetched at runtime — the compiled SHEET lacks the `ns`/`rating`/`firstMove`/`nav` fields they display; the solver's finishes are the authored `ns` texts with their leading rotations folded into the printed setup rotation). The compiled `js/sheet.js` + `data/classmap.json` currently have **no page consumer** — they remain the sheet pipeline's machine-checked data-quality gate (`npm run check`). `js/trainer.js` is the esbuild bundle of `src/trainer/` and no longer embeds the sheet. Display notation is normalized everywhere by the shared `engine.normAlg` so all surfaces show identical algs. Sheet keying: renderKey = `stateKey` (the full state — nothing is tip-fixable on a Skewb), canon = `realCanonKey` (y²-fold). There is no state-level 90° `y` symmetry (it swaps the corner tetrads), so a case's four viewing presentations pair at the **data level** via the JSON `direction` field, with the operational convention `prependAUF(p, frontAlg)` = alg for p = 0 Front / 1 Right / 2 Back / 3 Left ("case on the Right → rotate y → run the Front alg"); Front/Back share one canon and Right/Left the other (`caseStateOf` of a y²-prefixed alg equals the y²-sym image — engine-tested).

### The engine is the geometric core

`js/engine.js` (`window.OOEngine`) models the full no-tips Pyraminx state space (933,120 states): state model, moves, A4 rotations + LR mirror, canonicalization, the BFS-table optimal solver, alg parsing/notation, and the **keying + alg→case helpers** (`stateKey`, `realCanonKey`, `caseStateOf`, `algSolvesKey`, `normAlg`, …). State is `{ e:[piece,flip]×6, c:[L,R,B] twists, u: U twist }`. Render keys are edges + U-twist (`stateKey|uTwist`); center twists are tip-fixable and excluded from keys. The other layers are `render.js` (SVG net/3D diagrams, `window.OORender`), `account.js` (Firebase auth + per-user data, `window.OOAccount`), `navbar.js`, `config.js`, and `tables.js` (`window.OOTables`) — the shared IndexedDB cache + BFS dist/class-table builder that **both** `oo.js` (census) and `solver.js` use, with separate cache keys (`oo-dist-v1` for the shared distance table, `oo-classes-v2` for the census's canonical-class tables — v2 folds mirrors into the classes, 131,391) so the two value shapes can't collide. Load order on those pages is `engine.js` → `tables.js` → page script.

The keying helpers are genuinely shared now: the **Algorithms page and the trainer alias them off `window.OOEngine`** (`stateKey`, `realCanonKey`, `caseStateOf`, `prependAUF`, `enumFreeSlots`, `idx`/`unidx`) rather than carrying copies — so a change to move geometry or canonicalization in engine.js reaches every surface. trainer.html loads `js/engine.js` → `js/render.js` → `js/tables.js` before the bundle, so all three `window.*` layers exist at trainer eval; the trainer's dist table IS the census's (`OOTables.loadOrBuildDist`, IndexedDB key `oo-dist-v1`), and its lazy first-layer goal table is cached under the trainer-owned `trainer-fldist-v1`. What legitimately stays local to the trainer (`src/trainer/skewb-core.mjs` — plain ESM so `tools/test-trainer.mjs` can import it in Node) is the masked-scramble generator, the first-layer predicate/goal seeds, and the case model over the JSON — that's substrate, not keying.

### The sheet compiler (hermetic; Skewb re-key landed in M5)

`tools/compile-sheet.mjs` compiles the JSON through engine helpers only and writes two outputs: the `SHEET` data line of `js/sheet.js` (the file is read as a write-template to preserve its surrounding code) and `data/classmap.json` (canonical case key → lowercased subset id, generated from subset membership — never hand-maintain it). The carry-forward mechanism survives from upstream but starts empty: baseline = **committed snapshot `data/prior-sheet.json`** (currently `{}` — the JSON is the sole authority; re-baseline by overwriting it with a freshly built sheet if the JSON ever stops reproducing shipped cases you want kept), and the broken-alg allowlist `data/broken-algs.json` (currently `[]`; an array of `{renderKey, algorithm, reason}`, read by both the compiler and `check-sheet.mjs`). The compiler self-checks every emitted alg and refuses to write outputs that fail; a non-solving alg that isn't in the manifest fails both tools (`UNEXPECTED BROKEN`/`BROKEN`), a *new* unparseable alg fails the build, and an alg authored under a different case than it geometrically solves is an `ADVISORY`. `tools/check-sheet.mjs` re-validates the shipped sheet but shares the engine's keying functions, so it is not independent of engine-level keying bugs.

### Module strategy (why no `"type": "module"`)

The `js/*.js` files are **classic browser scripts** that attach to `window` (`OOEngine`, `OOSheet`, `OORender`, `OOAccount`). The Node tools (`.mjs`, ESM) load them for their **side effect**, not their return value: they set `globalThis.window = {}`, `require()` the file, then read `globalThis.window.OOEngine`. Each `js/` file wraps its body in an IIFE with a local `const module = {exports:{}}` that shadows Node's real `module`. Adding `"type": "module"` to package.json would break the tools' `require()` calls — it is intentionally omitted.

### Auth, roles, and the OO census

`account.js` exposes one shared sign-in session site-wide (Firebase Auth persists per-origin), with a `localStorage` "demo mode" fallback when `config.js` has no Firebase config. `config.js` `adminEmails` gates admin/moderator **UI only** — actual write authorization for the OO census (approving solutions, the public done-bitmap, the `moderators`/`moderatorInvites` collections) is enforced by **Firestore security rules**, version-controlled in `firestore.rules` (wired by `firebase.json`, deploy with `firebase deploy --only firestore:rules`). Admin in the rules is driven **solely** by an `admins/{uid}` collection — `adminEmails` is *not* trusted by the rules at all. These two notions of "admin" are decoupled, which is the main gotcha: an email in `adminEmails` makes the client render the Moderation tab, but if that signed-in account has no `admins/{uid}` document the rules deny every query the tab fires (surfaces as "Missing or insufficient permissions"). Bootstrap the first admin by creating `admins/{your-uid}` in the Firebase console (console writes bypass the rules); your uid is shown on the OO **About** page when signed in. See `SETUP.md` for the full setup/bootstrap walkthrough. Admin edits on the Algorithms page are a per-browser localStorage draft until **Export JSON** → commit → rebuild; there is no live shared store.
